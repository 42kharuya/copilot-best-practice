# CONTRIBUTING

このリポジトリは、**Core / Blueprints / Optional Packs** の3レイヤーで構成されています。

コントリビューション時は、まず「何を追加するか」より先に、**どのレイヤーに置くべきか**を判断してください。

## レイヤー別の保守方針

### Core

対象例:

- [AGENTS.md](AGENTS.md)
- [.github/copilot-instructions.md](.github/copilot-instructions.md)

保守方針:

- 最も強く保守する対象です
- 原則として、どのプロジェクトでも通用する内容だけを置きます
- 破壊的な方針変更は慎重に扱います
- 一時的な事情や特定技術依存の内容は入れません

期待する状態:

- 汎用的である
- 説明が短くても意味が通る
- 長期間古くなりにくい

### Blueprints

対象例:

- [.github/blueprints/README.md](.github/blueprints/README.md)
- `.github/blueprints/project-context/` 配下

保守方針:

- 実用性を高めるために積極的に改善する対象です
- ただし、特定プロジェクト専用の記述に寄りすぎないようにします
- 雛形として再利用しやすいことを優先します

期待する状態:

- コピーして使いやすい
- 何を書くべきかが判断しやすい
- プロジェクト固有情報の差し込み口として機能する

### Optional Packs

対象例:

- [.github/optional-packs/README.md](.github/optional-packs/README.md)
- `.github/optional-packs/case-studies/` 配下

保守方針:

- 参考価値は高いが、Core と同じ強度では保守しません
- 強い前提を持つ具体例を置く場所です
- 古くなりやすいため、検証状態や前提条件を明示します
- 本体の必須要件のように見えないように扱います

期待する状態:

- 何の条件で有効かがわかる
- 導入前後の違いや注意点がわかる
- 古くなった場合の扱いが明示される

## 保守優先度

| レイヤー | 保守優先度 | 方針 |
| :--- | :--- | :--- |
| **Core** | 最優先 | 汎用性と安定性を重視して強く保守する |
| **Blueprints** | 高 | 再利用しやすい差し込み口として継続的に改善する |
| **Optional Packs** | 中 | 参考価値を保ちつつ、前提条件と鮮度を明示する |

## deprecated の扱い

次のいずれかに当てはまる場合は、deprecated 扱いを検討します。

- 仕様変更で前提が崩れた
- より安全な置き場所へ移すべきと判断された
- 現在の推奨導線から外れた
- 実用例として誤解を招きやすくなった

deprecated にする場合の原則:

1. いきなり削除せず、まず deprecated であることを明示する
2. 代替の参照先がある場合は明記する
3. Core から Optional Packs へ移す場合は、理由を説明する
4. 完全に不要と判断できる場合のみ削除する

## 検証状態の表記ルール

必要に応じて、ドキュメントやサンプルに次の状態を明記します。

- **Verified**
  - 現時点で内容確認や検証が行われている
- **Needs Review**
  - 内容は残すが、最新仕様での再確認が必要
- **Example**
  - 参考例として掲載しており、そのまま適用を保証しない
- **Deprecated**
  - 非推奨。新規採用より代替案を優先する

基本方針:

- Core は可能な限り `Verified` を保つ
- Blueprints は `Verified` または `Example` が中心
- Optional Packs は `Example` や `Needs Review` を許容するが、状態を隠さない

## 更新ルール

### Core を更新するとき

- 本当に普遍原則かを先に確認する
- 技術依存や運用依存の内容を混ぜない
- 影響範囲が広いため、変更理由を明記する
- 可能なら関連ガイドも同時に確認する

### Blueprints を更新するとき

- 再利用しやすさを優先する
- 記入例が特定スタックへ寄りすぎていないか確認する
- 差し込みチェックリストや導線が壊れないようにする

### Optional Packs を更新するとき

- 前提条件を明記する
- 参考例なのか、検証済みなのかを明記する
- 本体へ戻すべき内容が混ざっていないか確認する

## コントリビューション時の確認項目

Pull Request 前に、次を確認してください。

- この変更は Core / Blueprints / Optional Packs のどこに属するか
- そのレイヤーに置く理由を説明できるか
- 古くなりやすい内容なら状態表記が必要ではないか
- 既存導線（README / `COPILOT_USAGE.md` / `about`）が必要に応じて更新されているか
- 本体の汎用性を壊していないか

## このリポジトリで重視すること

- 最新であることを隠さない
- 古くなった具体例を、Core の品質問題に見せない
- 汎用テンプレートとしての軸を保ちながら、実用例も提供する

迷った場合は、**より保守強度の低いレイヤーに一度置く**方が安全です。
